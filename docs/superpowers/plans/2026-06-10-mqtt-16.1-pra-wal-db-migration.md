# PR-A — WAL + DB-directory migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Put the prod SQLite DB on WAL inside a shared directory so PR-B's MQTT consumer can later be a second container safely reading/writing it — shipping no feature and proving the out-of-container readers against WAL first.

**Architecture:** Per-connection SQLite pragmas via a `connection_created` signal receiver (WAL + `synchronous=NORMAL` + `busy_timeout`); env-driven DB path so dev/prod differ by one env var; `web` mounts the DB *directory* (not the single file) so WAL's `-wal`/`-shm` siblings are shareable; the two host-side readers (`backup_db.py`, `ha_stats_export.py`) move to the new path and are verified against WAL in a backup-first prod runbook Claude drives around the merge.

**Tech Stack:** Django 6.0, SQLite, `python-decouple` `config()`, Docker Compose, `manage.py test`.

**Spec:** `docs/superpowers/specs/2026-06-10-mqtt-16.1-pra-wal-db-migration-design.md`

- Check: `~/.venvs/inventory/bin/python manage.py check`

Append to `inventory/tests.py`:

```python
import os
import sqlite3
import tempfile


class SqlitePragmaUnitTests(TestCase):
    """The receiver is exercised against a real *file* DB because WAL is not
    observable on an in-memory DB (which always reports journal_mode='memory')."""

    def test_pragmas_enable_wal_on_file_db(self):
        from inventory.db_pragmas import enable_sqlite_pragmas

        with tempfile.TemporaryDirectory() as d:
            raw = sqlite3.connect(os.path.join(d, "t.sqlite3"))

            class FakeConn:
                vendor = "sqlite"

                def cursor(self):
                    return raw.cursor()

            enable_sqlite_pragmas(sender=None, connection=FakeConn())

            cur = raw.cursor()
            self.assertEqual(
                cur.execute("PRAGMA journal_mode;").fetchone()[0].lower(), "wal"
            )
            self.assertEqual(cur.execute("PRAGMA synchronous;").fetchone()[0], 1)  # NORMAL
            self.assertEqual(cur.execute("PRAGMA busy_timeout;").fetchone()[0], 5000)
            raw.close()

    def test_receiver_is_noop_on_non_sqlite(self):
        from inventory.db_pragmas import enable_sqlite_pragmas

        called = {"cursor": False}

        class FakeConn:
            vendor = "postgresql"

            def cursor(self):
                called["cursor"] = True
                raise AssertionError("must not touch cursor on non-sqlite")

        enable_sqlite_pragmas(sender=None, connection=FakeConn())
        self.assertFalse(called["cursor"])
```

- [ ] **Step 2: Run test to verify it fails**

Run: `~/.venvs/inventory/bin/python manage.py test inventory.tests.SqlitePragmaUnitTests -v 2`
Expected: FAIL — `ModuleNotFoundError: No module named 'inventory.db_pragmas'`

- [ ] **Step 3: Write minimal implementation**

Create `inventory/db_pragmas.py`:

```python
"""SQLite PRAGMAs applied to every new database connection.

Enables WAL so the web app and (Phase 16.1) the telemetry consumer can read and
write the same SQLite file concurrently without `database is locked` errors.
Connected in ``InventoryConfig.ready()`` via the ``connection_created`` signal so
it runs identically for gunicorn workers, management commands, and the test
runner. A no-op on any non-SQLite backend.
"""


def enable_sqlite_pragmas(sender, connection, **kwargs):
    if connection.vendor != "sqlite":
        return
    cursor = connection.cursor()
    cursor.execute("PRAGMA journal_mode=WAL;")
    cursor.execute("PRAGMA synchronous=NORMAL;")
    cursor.execute("PRAGMA busy_timeout=5000;")
    cursor.close()
```

- [ ] **Step 4: Run test to verify it passes**

Run: `~/.venvs/inventory/bin/python manage.py test inventory.tests.SqlitePragmaUnitTests -v 2`
Expected: PASS (2 tests)

- [ ] **Step 5: Commit**

```bash
git add inventory/db_pragmas.py inventory/tests.py
git commit -m "feat: add SQLite WAL pragma receiver (Phase 16.1 PR-A)"
```

**FALLBACK if a reader fails** with `database is locked` / `unable to open database file` / `disk I/O error` (WAL `-shm` not writable by `jcoller` because the container wrote it as a different mapped uid):
  1. First try widening dir perms so `jcoller` can write `-shm`:
     `ssh inventory-manager 'chmod 0775 ~/inventory_db_dir && chmod 0664 ~/inventory_db_dir/inventory_db.sqlite3*'` — re-run the reader.
  2. If still failing, the durable fix is a follow-up commit dropping `?mode=ro` from the failing script's connect (it only issues SELECTs), opening it read-write so SQLite can manage `-shm`. Make that edit on a `fix/` branch, re-run, and note it on the PR. Do **not** leave the safety-net backup broken.
