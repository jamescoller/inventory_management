# PR-A — WAL + DB-directory migration (Phase 16.1, part 1 of 2)

**Date:** 2026-06-10
**Status:** approved (design); pending implementation
**Parent design:** [`docs/bambu-mqtt-integration.md`](../../bambu-mqtt-integration.md) §3
**Roadmap:** `todo.md` Phase 16.1 — "Infra (central)" bullet

## 1. Why

Phase 16.1 adds an MQTT telemetry consumer as a **second container** that reads and
writes the same SQLite DB as `web`. That makes it the first **sustained concurrent
writer** alongside gunicorn's workers. On the current setup — **no WAL** (`journal_mode`
defaults to `delete`) and the DB exposed as a **single bind-mounted file** — concurrent
writers produce intermittent `database is locked` → HTTP 500s.

This PR establishes the safe base **before** any consumer code exists, so the risky
prod-DB move happens once while only `web` touches the DB. It ships **no new feature** and
is independently revertible. PR-B (the consumer, `paho-mqtt`, telemetry models) builds on
top of it.

WAL also requires its `-wal`/`-shm` sibling files to be **shared across containers**.
Those siblings live next to the DB file; with today's single-*file* bind mount they would
be created inside each container's ephemeral filesystem (unshared) → divergent WAL state.
Mounting the **directory** is therefore mandatory, not cosmetic.

## 1a. Database engine decision — SQLite + WAL (not Postgres), with a trigger

Explicitly revisited 2026-06-10 (James) before building PR-A: **stay on SQLite + WAL.**

The 16.1 workload — 4 Bambu printers with **down-sampled** telemetry (<1 write/sec
sustained), a single-user web app, a 780 KB / 641-row DB — sits well inside SQLite's
comfort zone. The only real failure mode is web↔consumer lock contention, which WAL +
`busy_timeout` + a single writer process + short transactions resolve. None of the genuine
"migrate off SQLite" triggers (multiple un-serializable writers, sustained high write
throughput, network DB access from other hosts, multi-app-server, heavy analytics) apply at
homelab scale.

**Low lock-in makes deferring cheap.** The app is pure Django ORM; the *only* SQLite-specific
code in the repo is two helper scripts (`backup_db.py` online-backup API, `ha_stats_export.py`
direct read). A future move is an `ENGINE` change + `dumpdata`/`loaddata` (or `pgloader`) +
rewriting those two scripts — no escalating penalty for waiting.

**Migration trigger (write it down so it's not re-litigated):** if, after the consumer is
live, we observe **measurable** contention — `database is locked` in logs or 5xx traced to DB
locks — migrate to **PostgreSQL** (the stronger Django fit; not MariaDB). PR-A's WAL change is
not wasted in that event — it simply stops being load-bearing. Until then, Postgres stays
deferred (matches `docs/bambu-mqtt-integration.md` §3: "Postgres stays Trashed until
contention is measurable").

## 3. Design

### 3.1 WAL pragmas — `connection_created` signal
Apply per-connection pragmas via a `django.db.backends.signals.connection_created`
receiver (registered in the inventory app's `apps.py` `ready()`):

```python
# inventory/apps.py (ready) → connect a receiver
def _enable_sqlite_wal(sender, connection, **kwargs):
    if connection.vendor == "sqlite":
        with connection.cursor() as cur:
            cur.execute("PRAGMA journal_mode=WAL;")
            cur.execute("PRAGMA synchronous=NORMAL;")
            cur.execute("PRAGMA busy_timeout=5000;")
```

Rationale: multiple pragmas must run as separate statements; a single SQLite
`init_command` `cursor.execute(...)` runs only one statement, so the OPTIONS route is
unreliable here. The signal runs on every new connection (gunicorn worker, management
command, test runner) and is the canonical Django pattern. `journal_mode=WAL` is
persistent on the file; re-asserting per-connection is harmless. **Guarded on
`vendor == "sqlite"`** so it is inert if the DB ever changes.

### 3.2 Env-driven DB path
`settings.py` DATABASES:

```python
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.sqlite3",
        "NAME": config("SQLITE_DB_PATH", default=str(BASE_DIR / "inventory_db.sqlite3")),
    }
}
```

- **Dev (Claude LXC):** `SQLITE_DB_PATH` unset → default = repo-root `inventory_db.sqlite3`.
  Unchanged behaviour.
- **Prod (`~/.env_inventory` on the app LXC):** add one line
  `SQLITE_DB_PATH=/app/db/inventory_db.sqlite3`.

This keeps container paths out of `settings.py`; dev/prod differ only by that env line.
**Flag (CLAUDE.md `.env` rule):** new `config()` call → new `~/.env_inventory` entry. It is
a path, not a secret, but lives in the same uncommitted env file for consistency. Claude
adds the line during the runbook (§3.5).

### 3.3 Compose — directory mount
`docker-compose.yml`, `web` service:

```yaml
    volumes:
-     - ${HOME}/inventory_db.sqlite3:/app/inventory_db.sqlite3
+     - ${HOME}/inventory_db_dir:/app/db
      - static_volume:/app/staticfiles
```

The DB and its `-wal`/`-shm` siblings live in `~/inventory_db_dir/` on the host, mounted
at `/app/db` in the container; `SQLITE_DB_PATH` points at
`/app/db/inventory_db.sqlite3`. (PR-B's `telemetry` service mounts the same directory.)

### 3.4 Reader path updates + WAL `mode=ro` verification
Both out-of-container readers open the DB with `file:...?mode=ro`. A read-only open of a
**WAL** DB must be able to access the `-shm` shared-memory file; depending on the
ownership/permissions of the `-wal`/`-shm` written by the container, `mode=ro` can fail
("unable to open database file") or, worse, read a pre-WAL snapshot.

- **`scripts/backup_db.py`** — `DEFAULT_SRC` → `~/inventory_db_dir/inventory_db.sqlite3`.
  This is the **safety net**, so its `mode=ro` online-backup **must** be proven against the
  WAL DB in the runbook. The online-backup API yields a consistent snapshot under WAL when
  the source opens successfully.
- **`scripts/ha_stats_export.py`** — `DB_PATH` → new location; verify its `mode=ro` read.

**If `mode=ro` fails against WAL** (decided by the runbook test, not guessed): the minimal,
documented fallback is to relax the source open to a normal read-only-by-convention
connection (drop `?mode=ro`, keep SELECT-only) and/or ensure `-wal`/`-shm` are group/word
readable. The scripts default to these paths, and the crons call them with no path
argument, so **no cron edits are required**.

### 3.5 Migration runbook (Claude drives, coordinated with merge)
Executed over SSH as `jcoller` (no root: `~/` file ops + `docker compose` only). Downtime ≈
image build (~1–2 min).

**Claude owns the merge** (delegated by James 2026-06-10), which makes the ordering fully
deterministic — no hand-off race. Merging PR-A triggers the GH-Action deploy
(`git reset --hard origin/master` → `down` → `up -d --build`); if that `up` ran the *new*
compose while the DB was still at the old path, the container would create a fresh **empty**
DB at `/app/db`. Claude eliminates that by doing the prep (down + move + env) **before**
merging, so the deploy's `up` lands on the already-moved DB:

1. **Backup** the live DB (`backup_db.py`, old path) → confirm a fresh `.gz` on the NAS
   (app still up).
2. `docker compose down` (stop all writers — downtime starts).
3. `mkdir -p ~/inventory_db_dir && mv ~/inventory_db.sqlite3 ~/inventory_db_dir/`
   (no `-wal`/`-shm` yet — WAL not enabled pre-migration).
4. Add `SQLITE_DB_PATH=/app/db/inventory_db.sqlite3` to `~/.env_inventory`.
5. **Merge PR-A** (Claude). The deploy runs `down` (no-op) → `up -d --build` with the new
   compose on the moved DB; entrypoint runs `migrate` on the now-WAL DB, then gunicorn.
   (Claude may instead `up -d --build` the merged checkout directly to control timing — same
   result; pick whichever keeps downtime shortest.)
6. **Verify:** app responds; `PRAGMA journal_mode` returns `wal`; `-wal`/`-shm` siblings
   exist in `~/inventory_db_dir/`; row counts intact (**641 items / 107 locations / 264
   products**).
7. **Prove the readers against WAL:** run `backup_db.py` once (new path) → integrity-ok
   `.gz`; run `ha_stats_export.py` once → valid `inventory_stats.json`. Apply the §3.4
   fallback only if a reader fails.
8. Confirm the nightly backup cron + 5-min ha-stats cron still fire (they call the scripts
   with defaults).

Net downtime = step 2 → step 5's `up` completing. Because Claude controls prep + merge as one
sequence, this is a single uninterrupted window (~1–2 min), not gated on a hand-off.

**Merge readiness (when Claude merges):** implementation complete, local suite green
(incl. the audit canary), `manage.py check` clean, and a self/peer code review pass. James
does not gate the merge for PR-A but is notified before the prod runbook runs.

## 4. Testing

- **Local (TDD-friendly):**
  - A test asserting a fresh DB connection reports `PRAGMA journal_mode == "wal"` and
    `busy_timeout == 5000` (proves the signal fires under the test runner).
  - Full existing suite green on a WAL-enabled local DB — the **audit reconcile suite is
    the canary** (subtle locking/transaction regressions surface there first).
  - `manage.py check` clean.
- **Prod:** the runbook's verify + reader-proof steps (§3.5 #6–8) are the acceptance gate.

## 5. Acceptance criteria
- [ ] `journal_mode=WAL`, `busy_timeout=5000`, `synchronous=NORMAL` active on prod.
- [ ] DB lives in `~/inventory_db_dir/`; `web` mounts the directory; `SQLITE_DB_PATH` set
      in `~/.env_inventory`.
- [ ] Prod data intact (641 / 107 / 264) and app healthy post-migration.
- [ ] `backup_db.py` produces a valid, integrity-checked snapshot **from the WAL DB**.
- [ ] `ha_stats_export.py` produces valid JSON from the WAL DB.
- [ ] Dev unaffected (default path, suite green).

## 6. Rollback
Revert `settings.py` + `docker-compose.yml`, remove the `.env` line, `docker compose down`,
`mv ~/inventory_db_dir/inventory_db.sqlite3 ~/inventory_db.sqlite3` (checkpoint first so no
`-wal` data is stranded: `sqlite3 … "PRAGMA wal_checkpoint(TRUNCATE);"` or simply let the
clean shutdown checkpoint), `docker compose up -d`. A backup is taken immediately prior
(runbook #1). WAL is forward/backward compatible after a checkpoint.

## 7. Risks
- **`mode=ro` vs WAL** (readers) — primary risk; mitigated by proving both readers in the
  runbook with a known minimal fallback (§3.4). De-risked by ordering this *before* the
  consumer.
- **Stranded `-wal` on rollback** — mitigated by checkpoint-on-shutdown / explicit
  `wal_checkpoint(TRUNCATE)`.
- **Auto-deploy timing** — the GH-Action deploy hard-resets to `origin/master` and runs
  `down/up`; the one-time `mv` + `.env` edit must precede the new compose coming up. **Fully
  controlled:** Claude owns the merge, so it performs steps 1–4 before merging (step 5) — no
  race window.
